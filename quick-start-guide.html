üéØ Getting Started

1 Initial Setup

If this is your first time, you'll need to configure your Supabase database connection. The system will prompt you for your Supabase Project URL and Anon Key. You can find these in your Supabase project settings under "API".

2 Create Database Tables

Click "Show Database Setup Instructions" to get the SQL commands needed to create the required tables in your Supabase database. Copy these commands and run them in your Supabase SQL Editor (found in the left sidebar of your Supabase dashboard).

3 Sign In with Google

The system uses Google authentication to track who creates, reviews, and approves items. Click "Sign in with Google" to authenticate. Your email will be used to identify you in comments and audit logs.

4 Load Sample Data (Optional)

To explore the system with realistic data, download the sample_data.sql file and run it in your Supabase SQL Editor. This creates a complete mobile banking app project with requirements, test cases, defects, and relationships that demonstrate all system features.

5 Create Your First Project

Click the project dropdown at the top and select "+ New Project". Give it a name, project ID (PID), and optionally assign a project manager and lead engineer. All items you create will belong to the currently selected project.

Requirement Levels (Hierarchical Decomposition)

Requirements can be organized into seven hierarchical levels, each representing increasing detail:

  • System - Highest level, describes what the entire system must do
  • Sub-System - Major subsystems or components
  • Assembly - Logical groupings within subsystems
  • Sub-Assembly - Detailed assemblies
  • Component - Individual components
  • Sub-Component - Parts of components
  • Material - Lowest level, material specifications
üí° Tip: Use epics for high-level organization (like features or themes), then nest requirements underneath them. The visual indentation in the tree shows the requirement level automatically.

üîÑ Common Workflows

Creating Requirements

  1. Click "New Item" button
  2. Select type (epic, requirement, test-case, or defect)
  3. Fill in title and description (required)
  4. For requirements: add rationale and select the appropriate level
  5. Optionally assign an owner and reviewer
  6. Click "Save"

Review & Approval Process

  1. Create a requirement and assign a reviewer
  2. When ready for review, click "üì§ Send for Review"
  3. The reviewer will see it in their "Items Awaiting Your Review" panel
  4. Reviewer opens the item and clicks "Approve" or "Reject"
  5. If rejected, item returns to draft for revision
  6. If approved, requirement is locked until edited (which resets it to draft)

Test Case Execution

  1. Create a test case and assign a tester
  2. Link it to the requirement(s) it tests using the "Link Items" button
  3. When ready for testing, click "üß™ Ready for Test"
  4. The tester will see it in their "Tests Awaiting Execution" panel
  5. Tester opens the item and clicks "Pass" or "Fail"
  6. If failed, tester can create a defect and link it to the requirement

Creating Relationships

Relationships show how items connect. To create a relationship:

  1. Click "Link Items" button
  2. Select the source item (From)
  3. Select relationship type:
    • tests - Test case validates a requirement
    • derives-from - Lower-level requirement decomposes from higher-level
    • depends-on - Implementation depends on another requirement
    • relates-to - General connection (like defect to requirement)
  4. Select the target item (To)
  5. Click "Add Relationship"

üí° Power User Tip: Combine filters for powerful queries. For example:
  • Type: test-case + Status: failed = Show only failing tests
  • Type: defect + Status: backlog = Show deferred bugs
  • Type: requirement + Status: in-review = Show items awaiting review

Detail Panel

Click any item in the tree to open the detail panel on the right. Here you can:

  • View all details - Description, rationale, status, priority, owner, etc.
  • Edit the item - Click the edit icon (pencil)
  • See relationships - View in list or graph mode; click relationships to navigate
  • Add comments - Collaborate with team members
  • View audit history - See complete lifecycle and all changes
  • Execute workflow actions - Send for review, mark ready for test, approve/reject, pass/fail

Drag & Drop

üé® Best Practices

Project Organization

  • Create separate projects for different products, releases, or major initiatives
  • Use meaningful Project IDs (PIDs) like "MOB-2" or "WEB-1" for easy reference
  • Assign project managers and lead engineers for accountability

Requirements Hierarchy

  • Start with epics to define major features or themes
  • Write system-level requirements first to define high-level capabilities
  • Decompose into lower levels only as needed for clarity
  • Use "derives-from" relationships to show parent-child requirements
  • Write requirements as "System shall..." or "Component shall..." statements

Test Coverage

  • Create at least one test case for every requirement
  • Link test cases to requirements using "tests" relationships
  • Include detailed test methods - use URLs to link to automation frameworks
  • Use the Reports view to identify requirements lacking test coverage

Workflow Discipline

  • Always assign reviewers to requirements that need approval
  • Send items for review when they're ready, don't skip the approval process
  • Use comments to discuss changes rather than just making silent edits
  • When tests fail, create defects and link them to the relevant requirements
  • Update defect status as you work on fixes (not-started ‚Üí in-process ‚Üí resolved)

Collaboration

  • Use comments to ask questions, provide feedback, and document decisions
  • Check audit history to understand why changes were made
  • Assign owners to ensure accountability for each item
  • Use the review panels to stay on top of items awaiting your action

‚ùì Troubleshooting

Common Issues

Items not appearing in tree view

  • Check that you have the correct project selected
  • Clear any type or status filters that might be hiding items
  • Verify the items belong to the selected project by checking in Supabase

Can't approve/reject items

  • Only the assigned reviewer can approve/reject requirements
  • Only the assigned tester can pass/fail test cases
  • Item must be in "in-review" or "ready-for-test" status

CSV import errors

  • Ensure type is exactly: epic, requirement, test-case, or defect
  • Verify title is not empty
  • Check that status values match the valid statuses for that item type
  • Make sure you have a project selected before importing
